# Запрос разрешения на Push

**ID:** LOGIC-06  
**Тип:** Логика  
**Домен:** 09. Логики  
**Приоритет:** Medium  
**Статус:** Актуален  
**Функциональные блоки:** FB-PUSH-001

---

## Обзор

Логика запроса разрешения на отправку push-уведомлений и регистрации push-токена на сервере.

### User Story

> Как пользователь, я хочу получать уведомления о состоянии брони и напоминания.

---

## Флоу

```mermaid
flowchart TD
  Start[Открытие экрана] --> Prompt[Показать rationale (если нужно)]
  Prompt --> RequestPermission[Запрос системного разрешения]
  RequestPermission -->|Granted| RegisterAPI[POST /devices/register]
  RequestPermission -->|Denied| StorePreference[Сохранить отказ]
```

---

## Подход

- Показывать объяснение (rationale) перед системным диалогом, если отказ был ранее.
- После получения push токена отправлять его на сервер вместе с deviceuuid и user_id.
- При отказе сохранять факт отказа и предлагать включить в настройках.

---

## API запросы

### POST /devices/register

**Body:** { "deviceuuid": "..", "push_token": "..", "platform": "ios|android", "user_id": ".." }

**Обработка ответа:**

| Код | Действие |
|-----|----------|
| 200 | Устройство зарегистрировано |
| 409 | Уже зарегистрировано — обновить запись |

---

## Локальное хранение

| Ключ | Тип хранения | Описание |
|------|--------------|----------|
| `push_permission` | Локальный кэш | granted/denied/not_asked |

## Критерии приёмки

| ID | Критерий |
|----|---------|
| AC-001 | После разрешения push токен отправлен и зарегистрирован на сервере |
| AC-002 | При отказе пользователь может позже включить уведомления в настройках |

## Обработка ошибок

| Тип ошибки | Контекст | Действие |
|------------|----------|----------|
| Отсутствие сети | Регистрация токена | Сохранить локально и retry при следующем запуске |
