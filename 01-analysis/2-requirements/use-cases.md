# Use cases / сценарии

> Этап 2. Сценарии использования. Обязательно прорабатываем **альтернативные потоки** —
> ИИ любит их забывать.
>
> **Скоуп: клиентское приложение и API для него.** Слоты, конфигурации трасс и маршалы поступают
> из существующего бэкенда через API (read-only).

## UC-1: Запись на заезд (себя и гостей)

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован; в расписании есть хотя бы один слот со свободными местами.
- **Триггер:** Клиент хочет забронировать место на заезд.

> **Бизнес-решение по составу гостей (R-013).** По гостям собираем только **количество мест и выбор экипировки** (своя / напрокат). Имена, контакты, возраст и иные данные гостей в MVP **не собираются**; ответственность за гостей несёт клиент, оформивший бронь.
>
> **Бизнес-решение по идемпотентности (R-022).** Клиент **обязан** передавать `Idempotency-Key` при создании брони. Повтор той же операции (тот же ключ и то же тело запроса) даёт идентичный ответ и не создаёт дубль брони. Повтор с тем же ключом, но другим телом — ошибка.
>
> **Время и источник истины (R-021).** Время хранится и передаётся в UTC, отображается в локальной зоне центра. Сервер — источник истины по времени и статусам; при расхождении клиентского прогноза и серверного ответа применяется серверный статус.

### Основной поток
1. Клиент открывает список слотов.
2. (Опц.) Применяет фильтры: дата/период, конфигурация трассы (короткая/длинная), наличие мест, маршал.
3. Открывает карточку слота и просматривает параметры (дата/время старта, **время прибытия = старт − 15 мин**, конфигурация трассы, маршал, цена, свободно мест, доступность прокатной экипировки, адрес центра и место сбора).
4. Указывает число мест (себя + до 2 гостей). По гостям указывается только количество (R-013).
5. Для каждого места выбирает вариант экипировки: своя (шлем + подшлемник) / напрокат.
6. Подтверждает запись. Клиент формирует и передаёт `Idempotency-Key` для этой операции (R-022).
7. Система проверяет лимит мест `max_seats = min(free_seats, track_config.capacity_cap, 3)` (три фактора: свободные места, потолок конфигурации трассы — короткая ≤8, длинная ≤14, лимит 3 на бронь) и **отдельно** прокатный фонд экипировки `rental_count ≤ free_rental_equipment`, создаёт запись, уменьшает счётчики свободных мест/прокатных комплектов.
8. Система показывает подтверждение с итоговой ценой (`price_total`, оплата офлайн) и добавляет запись в «Мои бронирования».

### Альтернативные потоки
- **A1. Вся экипировка — своя.** На шаге 5 прокатный фонд не затрагивается; уменьшается только число мест в группе.
- **A2. Часть мест — прокат.** На шаге 7 проверяется и уменьшается прокатный фонд только на число «прокатных» мест.
- **A3. Фильтр не дал результатов.** На шаге 2 список пуст — система предлагает сбросить/изменить фильтры.
- **A4. Новичковая конфигурация (короткая трасса).** Если клиент выбрал слот с короткой трассой, система учитывает потолок 8 мест (вместе с гостями); при попытке забронировать больше — срабатывает ограничение FR-13.

### Исключения / ошибки
- **E1. Не хватает мест в группе.** Запрошено больше, чем свободно мест → запись отклоняется, показывается актуальное число свободных мест.
- **E2. Не хватает прокатной экипировки.** Прокатных мест запрошено больше, чем свободно комплектов → система предлагает уменьшить число прокатных мест или выбрать «своя экипировка».
- **E3. Гонка запросов / слот заполнился параллельно.** На шаге 7 места уже заняты другим клиентом → запись отклоняется без овербукинга и двойной брони (NFR-8), список обновляется.
- **E4. Сетевой сбой / неопределённый результат при подтверждении (R-020, R-022).** Запись не создаётся частично; при неопределённом результате клиент повторяет операцию с **тем же** `Idempotency-Key`, повтор не приводит к задвоению (NFR-9). Мутации (создание брони) офлайн запрещены — при отсутствии сети показывается понятное сообщение, кнопка подтверждения недоступна.
- **E5. Слот отменён или уже стартовал (R-023).** Слот к моменту подтверждения отменён центром или заезд начался → запись отклоняется с соответствующим кодом (`slot_cancelled` / `slot_started`).
- **E6. Попытка записаться на уже отменённый центром слот (R-008).** Сервер возвращает HTTP 410 (`slot_cancelled`) с причиной; кнопка записи неактивна, клиенту показано пояснение.

### Матрица ошибок ключевых операций (R-023)

Канон таксономии — коды API. Источник локализованного текста — клиентское приложение (по коду ошибки и полям `details`).

| Код | HTTP | Обязательные `details` | UX-реакция | Retry |
| :-- | :-- | :-- | :-- | :-- |
| `slot_full` | 409 | `available_seats`, `available_rental_equipment` | Показать актуальные свободные места/прокатные комплекты, обновить карточку | Нет (изменить число мест) |
| `double_booking` | 409 | `booking_id` | Сообщить, что бронь на слот уже есть; перейти в «Мои бронирования» | Нет |
| `slot_cancelled` | 410 | `slot_id`, `reason` | Сообщить об отмене слота центром с причиной, убрать CTA записи | Нет |
| `slot_started` | 422 | `slot_id`, `start_at` | Сообщить, что заезд начался/прошёл, запись недоступна | Нет |
| `invalid_code` | 400/401 | — | Сообщение «неверный код», поле OTP остаётся активным | Да (ввод заново) |
| 401 (unauthorized) | 401 | — | Очистка токена, переход на авторизацию (R-016) | Да (после входа) |
| 403 (forbidden) | 403 | — | Сообщение об отсутствии прав; действие недоступно | Нет |
| 429 (rate limit) | 429 | `retry_after` | Сообщение «слишком много попыток», отложенный повтор | Да (после `retry_after`) |
| 5xx (server error) | 5xx | — | Единый Error/Retry; для брони — повтор по тому же `Idempotency-Key` | Да |

### Постусловия
- При успехе: создана запись клиента; счётчики слота и прокатного фонда уменьшены; запись видна в «Мои бронирования».
- При ошибке: состояние слота не изменилось.

## UC-2: Отмена записи

- **Актёры:** Клиент
- **Предусловия:** У клиента есть предстоящая запись; заезд ещё не стартовал.
- **Триггер:** Клиент решает отменить участие.

> **Бизнес-решение по объёму отмены (R-014).** Отмена выполняется **только целиком**: отменяется вся бронь со всеми местами и прокатными комплектами экипировки. Частичная отмена (например, одного гостя) **не поддерживается**; при ранней отмене освобождаются все места и прокатная экипировка брони.
>
> **Время и источник истины (R-021).** Время хранится и передаётся в UTC, отображается в локальной зоне центра. Сервер — источник истины по времени и по статусу отмены. Граница политики: **ровно 3 часа до старта = ещё ранняя (бесплатная) отмена**. При расхождении клиентского прогноза (ранняя/поздняя) и серверного ответа применяется **серверный** статус.

### Основной поток
1. Клиент открывает «Мои бронирования» и выбирает предстоящую запись.
2. Нажимает «Отменить» (отмена всей брони целиком, R-014).
3. Сервер определяет время до старта (источник истины — серверное время, R-021).
4. **Ранняя отмена (≥3 ч до старта):** вся бронь помечается «отменена»; **все** забронированные места и прокатные комплекты экипировки возвращаются в слот.
5. Система показывает подтверждение отмены.

### Альтернативные потоки
- **A1. Поздняя отмена (<3 ч до старта).** На шаге 4 запись помечается «поздняя отмена» (`late_cancel`); места НЕ освобождаются; денежных штрафов нет; клиенту показывается пояснение политики 3 часов.
- **A2. Расхождение клиентского прогноза и сервера (R-021).** Клиент показал «ранняя отмена», а сервер вернул «поздняя» (или наоборот) → применяется и отображается серверный статус.
- **A3. Бронь уже отменена центром (R-008).** Если слот был отменён центром до действия клиента, кнопка «Отменить» недоступна — бронь уже в статусе «Отменён центром» с причиной.

### Исключения / ошибки
- **E1. Заезд уже стартовал/прошёл (`slot_started`).** Кнопка отмены недоступна; запись нельзя отменить.
- **E2. Запись уже отменена (`already_cancelled`).** Повторная отмена не выполняется; показывается текущий статус.
- **E3. Отмена офлайн / неопределённый результат (R-020).** Мутация отмены офлайн запрещена — при отсутствии сети показывается понятное сообщение, кнопка недоступна; при неопределённом результате применяется единый Error/Retry, финальный статус берётся с сервера. Коды ошибок и UX-реакции — по матрице ошибок UC-1 (R-023).

### Постусловия
- Статус всей брони обновлён («отменена» / «поздняя отмена»).
- При ранней отмене все места и прокатные комплекты брони снова доступны другим клиентам.

## UC-3: Просмотр и фильтрация списка слотов

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован.
- **Триггер:** Клиент хочет найти подходящий заезд.

### Основной поток
1. Клиент открывает список слотов. По умолчанию (`only_available=false`) показываются слоты на **ближайшие 7 дней** (дефолт API: `date_from = now`, `date_to = now + 7 дней`; больший период — явным фильтром дат, R-027), отсортированные по времени старта (по возрастанию).
2. Заполненные (`free_seats = 0`) и отменённые центром слоты отображаются в списке с пометкой «Мест нет» и неактивной (disabled) CTA записи (см. R-012).
3. (Опц.) Задаёт фильтры: дата/период, конфигурация трассы (короткая/длинная), наличие мест, маршал.
4. Система показывает отфильтрованный список с ключевыми параметрами каждого слота, сохраняя сортировку по времени старта.

### Альтернативные потоки
- **A1. Сброс фильтров.** Клиент очищает фильтры → возвращается дефолтная выдача (слоты на ближайшие 7 дней, дефолт без фильтров).
- **A2. Только свободные слоты.** Клиент включает фильтр наличия мест (`only_available=true`) → заполненные/отменённые слоты скрываются.
- **A3. Семантика комбинирования фильтров (R-026).** Внутри одной группы значения комбинируются по «ИЛИ» (`track_config`, `marshal_id` — массивы), между разными группами — по «И». Границы периода дат включительные; ввод и трактовка дат — в локальной зоне центра. Активные фильтры отображаются клиенту.
- **A4. Пустой справочник маршалов.** Если список маршалов пуст, фильтр по маршалу скрывается.
- **A5. Быстрые пресеты дат.** Клиент выбирает «Сегодня / Эта неделя / Выходные» → система маппит пресет в `date_from`/`date_to` и обновляет выдачу.

### Исключения / ошибки
- **E1. Нет слотов под условия.** Показывается пустое состояние с подсказкой изменить/сбросить фильтры.
- **E2. Нет расписания вовсе.** В горизонте выдачи нет ни одного предстоящего слота → показывается пустое состояние «Заездов пока нет».
- **E3. Просмотр офлайн (R-020).** При отсутствии сети показывается ранее загруженный кэш списка с пометкой «данные могли устареть»; единый Error/Retry для повторной загрузки.

### Постусловия
- Клиент видит актуальный (или помеченный как устаревший) список предстоящих слотов; данные расписания не изменяются.

## UC-4: Авторизация по SMS OTP

- **Актёры:** Клиент
- **Предусловия:** Приложение установлено; у клиента есть действующий номер телефона с возможностью получения SMS.
- **Триггер:** Клиент впервые открывает приложение или его сессия истекла.

> **Бизнес-решение по способу входа (FR-43).** Подтверждение личности — **SMS OTP** (одноразовый код в SMS). Пароли не используются.
>
> **Безопасность сессии (NFR-18).** После успешного входа сервер выдаёт пару JWT: короткоживущий `access` (~15 мин) и долгоживущий `refresh` (~30 дней). Оба хранятся в защищённом хранилище ОС (Keychain/Keystore).
>
> **Антифрод OTP (NFR-19).** Отправка кода ограничена rate-limit'ами (~1 запрос в 30–60 с на номер, ~5 в час, ≤5 попыток ввода до временной блокировки). Код имеет конечный TTL.

### Основной поток
1. Клиент открывает экран входа и вводит номер телефона (нормализованный формат, +7…).
2. Нажимает «Получить код». Система отправляет запрос на `/auth/otp/send` (с учётом rate-limit'ов, NFR-19).
3. Сервер генерирует OTP с TTL и отправляет SMS. Клиенту показывается экран ввода кода с таймером обратного отсчёта.
4. Клиент вводит код из SMS.
5. Система отправляет `/auth/otp/verify`. При успехе сервер возвращает пару токенов (`access` + `refresh`).
6. Клиент сохраняет токены в Keychain/Keystore (NFR-18) и переходит на главный экран (список слотов).
7. Если у клиента ещё нет профиля — сервер создаёт минимальный профиль (имя + телефон); при необходимости приложение запрашивает имя.

### Альтернативные потоки
- **A1. Повторная отправка кода.** Клиент нажимает «Отправить код повторно» после истечения таймера → новый код с новым TTL; предыдущий код инвалидируется.
- **A2. Обновление access-токена (401-flow, NFR-18).** При получении `401` от API клиент автоматически пытается обновить `access` по `refresh`; при успехе продолжает операцию без участия пользователя.
- **A3. Истёк refresh.** При невозможности обновить `access` по `refresh` клиент очищает хранилище и переводит пользователя на экран входа (перелогин через OTP).

### Исключения / ошибки
- **E1. Rate limit на отправку (429).** Показывается сообщение «Слишком много попыток, повторите через N секунд» (поле `retry_after`).
- **E2. Неверный код (`invalid_code`).** Поле OTP остаётся активным; счётчик попыток уменьшается; при исчерпании — временная блокировка с понятным сообщением.
- **E3. SMS не дошло.** Клиент ждёт и запрашивает повторную отправку (A1); при системных проблемах SMS-провайдера — деградация сервиса (NFR-23), вход временно недоступен.
- **E4. Сетевой сбой при отправке/проверке.** Единый Error/Retry; при неопределённом результате — повтор запроса без изменения состояния (проверка OTP идемпотентна по коду).

### Постусловия
- При успехе: клиент авторизован, токены сохранены в защищённом хранилище, профиль доступен.
- При ошибке: токены не выданы, клиент остаётся на экране входа.

## UC-5: Просмотр деталей слота и места сбора

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован; в списке выбран конкретный слот.
- **Триггер:** Клиент хочет уточнить детали заезда и посмотреть, где собираться.

> **Интеграция с Яндекс.Картами (NFR-26, FR-44).** Координаты точки встречи (`meeting_point`) обязательны. API-ключ хранится как параметр конфигурации окружения. При недоступности картографического сервиса или превышении квоты — **fallback-режим без карты**: адрес/координаты текстом.

### Основной поток
1. Клиент открывает карточку слота.
2. Система отображает: дату/время старта, **время прибытия** (старт − 15 мин), конфигурацию трассы и её описание, маршала, цену, общее и свободное число мест, доступность прокатной экипировки, адрес центра.
3. Система загружает и отображает карту с точкой `meeting_point` через Яндекс.Карты.
4. Клиент изучает детали и принимает решение о записи или возврате к списку.

### Альтернативные потоки
- **A1. Fallback карты (NFR-26).** При недоступности сервиса карт или превышении квоты система показывает текстовое название точки и координаты без интерактивной карты; бронирование продолжает работать.
- **A2. Слот заполнен / отменён.** В карточке показывается пометка «Мест нет» или «Отменён центром» с причиной; CTA записи неактивна.

### Исключения / ошибки
- **E1. Не удалось загрузить детали слота.** Показывается единый Error/Retry (NFR-24); клиент может вернуться к списку.
- **E2. Слот исчез из расписания (отменён между просмотром списка и карточки).** Сервер возвращает `slot_cancelled` (410) — показывается причина отмены.

### Постусловия
- Клиент ознакомился с деталями; состояние системы не изменилось.

## UC-6: Получение push-уведомления

- **Актёры:** Клиент (пассивный получатель), Система (инициатор)
- **Предусловия:** Клиент дал разрешение на push-уведомления; у него есть предстоящая бронь.
- **Триггер:** Наступил тайминг напоминания или центр отменил заезд.

> **Канал и тайминг (NFR-17, FR-33).** Канал MVP — системный push (APNs/FCM). Напоминания приходят **за 24 ч и за 3 ч** до старта (`reminder_hours = [24, 2]`); конкретные часы клиент берёт из ответа `createBooking`, не хардкодит.
>
> **Отмена центром (R-008, FR-46).** При отмене заезда центром (в т.ч. по погоде — сильный дождь, гололёд) клиент получает push с причиной; бронь переходит в статус «Отменён центром».

### Основной поток (напоминание)
1. Сервер по расписанию (`reminder_hours`) инициирует отправку push через APNs/FCM.
2. Устройство доставляет push; клиент видит уведомление «Напоминание: заезд через N, адрес центра».
3. По тапу на уведомление приложение открывается на экране «Мои бронирования» с нужной записью.

### Альтернативные потоки
- **A1. Отмена центром (R-008).** Сервер инициирует push «Заезд отменён: <причина>». Бронь в приложении переходит в статус «Отменён центром» с причиной; CTA записи на этот слот становится неактивной.
- **A2. Клиент отозвал разрешение на push.** Напоминания не доставляются; приложение штатно работает, информация о бронях доступна в «Мои бронирования»; при необходимости приложение может подсказать включить уведомления.
- **A3. Невалидный push-токен.** Сервер прекращает отправку без ошибки для пользователя; при следующем входе приложение регистрирует новый токен.

### Исключения / ошибки
- **E1. Недоступность push-сервисов (APNs/FCM).** Деградация сервиса (NFR-23): бронирование и отмена работают, напоминания не доставляются.
- **E2. Приложение удалено с устройства.** Push не доставляется; сервер фиксирует невалидный токен и прекращает отправку.

### Постусловия
- При успехе: клиент получил напоминание или уведомление об отмене.
- Состояние брони синхронизировано с сервером при следующем открытии приложения.

## UC-7: Управление профилем и удаление аккаунта

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован.
- **Триггер:** Клиент хочет изменить данные или удалить аккаунт.

> **ПДн и удаление (NFR-20, FR-48).** При удалении аккаунта активные брони аннулируются (освобождают места и прокатную экипировку); прошедшие брони анонимизируются и сохраняются для истории/отчётности (без привязки к ПДн удалённого клиента). Retention прошедших броней — 12 месяцев, далее анонимизация.

### Основной поток (редактирование профиля)
1. Клиент открывает экран профиля.
2. Редактирует имя и/или номер телефона.
3. Сохраняет изменения → сервер обновляет профиль; приложение отображает актуальные данные.

### Альтернативные потоки
- **A1. Выход из аккаунта (FR-49).** Клиент нажимает «Выйти» → приложение очищает токены из Keychain/Keystore и переходит на экран входа.
- **A2. Удаление аккаунта (FR-48).** Клиент нажимает «Удалить аккаунт» → приложение показывает диалог подтверждения с описанием последствий (аннулирование активных броней, анонимизация истории). При подтверждении сервер:
   - аннулирует активные брони (освобождает места и прокатную экипировку);
   - помечает профиль как удалённый;
   - анонимизирует прошедшие брони;
   - отзывает все активные токены.
   Приложение очищает локальное хранилище и переходит на экран входа.

### Исключения / ошибки
- **E1. Есть активные брони при удалении.** Система предупреждает, что все активные брони будут аннулированы; клиент подтверждает явно.
- **E2. Сетевой сбой при сохранении/удалении.** Единый Error/Retry; при неопределённом результате удаления — повтор запроса, финальный статус берётся с сервера.
- **E3. Попытка редактирования после отзыва сессии.** Сервер возвращает `401` → срабатывает 401-flow (NFR-18), клиент переходит на экран входа.

### Постусловия
- При редактировании: профиль обновлён.
- При выходе: локальные токены очищены.
- При удалении: аккаунт деактивирован, активные брони аннулированы, история анонимизирована, локальные данные очищены.
