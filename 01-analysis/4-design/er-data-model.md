# Модель данных

> Этап 3. Проектирование. Описание сущностей, атрибутов и связей + черновик ERD.
>
> **Скоуп: клиентское приложение и API для него.** Это **ресурсная модель API** (что клиент
> читает/создаёт), а не схема БД, которую мы проектируем: хранение и бизнес-логика принадлежат
> **существующей инфраструктуре**.
>
> - **TrackConfig, Marshal, Slot** — read-only-проекция ресурсов существующего бэкенда; приходят
>   через API, клиент их не создаёт и не редактирует.
> - **Client, Booking** — ресурсы, которыми оперирует клиентский API (регистрация и бронирования).
> - Сущности оценок/рейтингов в скоуп не входят (обеспечиваются существующей инфраструктурой).
> - **Данные существующей инфраструктуры (R-015).** Проект учебный/тестовый, легаси-данных нет:
>   эта модель (вместе с `rental_price`, `meeting_point`, `free_rental_equipment`)
>   считается **канонической** и совпадает с контрактом API. Миграция/backfill и поведение при
>   отсутствии полей не рассматриваются — бэкенд по условию отдаёт все поля модели.

## Сущности и атрибуты

### Client (Клиент) — **ИЗМЕНЯЕТСЯ ПРИЛОЖЕНИЕМ**

| Атрибут | Тип | Описание |
| :-- | :-- | :-- |
| id | UUID (PK) | Идентификатор клиента |
| name | string | Имя (до 50 символов) |
| phone | string (unique) | Номер телефона — логин; вход подтверждается кодом из SMS (OTP) |
| created_at | datetime | Дата регистрации |
| deleted_at | datetime? | Дата удаления (null, если не удалён) |

> **Вход/регистрация** — трёхшаговый поток (телефон → код из SMS → имя для нового пользователя),
> см. [SCR-001](../3-design-brief/scr-001-registration.md). Сам код подтверждения (OTP) и его
> проверка — на стороне бэкенда, отдельной сущностью в модели не хранится. Смена телефона в
> профиле также подтверждается кодом.

> **История броней и удаление аккаунта (R-025, NFR-20).** Клиенту доступна **вся история** своих
> броней (активные, отменённые, поздние, отменённые центром, прошедшие) — `listBookings`
> отдаёт её **постранично** (пагинация, `limit`/`offset`).
>
> **Переход состояний при `deleteAccount`.** При удалении аккаунта:
> - **Активные брони** (`status = active`) переводятся в `cancelled` (системная отмена):
>   проставляется `cancelled_at`, **освобождаются места и прокатная экипировка** в слоте — как при
>   обычной ранней отмене (`free_seats += seats_count`, `free_rental_equipment += rental_count`).
> - **Прошедшие / завершённые брони** сохраняются обезличенными (для учёта).
> - **ПДн клиента** (`name`, `phone`) **анонимизируются**, а не удаляются жёстко; `Client.phone`
>   **освобождается** для повторной регистрации.
> - **История** связанных броней **сохраняется обезличенной** (для учёта/статистики
>   существующей инфраструктуры).

### TrackConfig (Конфигурация трассы) — **READ-ONLY (из бэкенда)**

| Атрибут | Тип | Описание |
| :-- | :-- | :-- |
| id | UUID (PK) | Идентификатор конфигурации |
| name | string | Название (например, "Короткая трасса") |
| description | string? (nullable) | Описание трассы для карточки слота (SCR-003). Опциональный; может отсутствовать или быть `null` (до 300 символов) |
| type | enum (`short`/`long`) | Тип: `short` (новички) / `long` (опытные) |
| capacity_cap | int | Потолок мест (короткая ≤8, длинная ≤14) |
| duration_min | int | Длительность заезда, мин (≈15–20) |

> **Справочник конфигураций трасс.** Приложение не создаёт и не редактирует конфигурации —
> только потребляет их через API для фильтрации и отображения.

---

### Marshal (Маршал) — **READ-ONLY (из бэкенда)**

| Атрибут | Тип | Описание |
| :-- | :-- | :-- |
| id | UUID (PK) | Идентификатор маршала |
| name | string | Имя маршала (до 50 символов) |
| photo_url | string? (nullable) | URL фото (опционально) |

> **Справочник маршалов-инструкторов.** Приложение не создаёт и не редактирует маршалов —
> только потребляет их через API для фильтрации и отображения.

---

### Slot (Слот / Заезд) — **READ-ONLY (из бэкенда)**

| Атрибут | Тип | Описание |
| :-- | :-- | :-- |
| id | UUID (PK) | Идентификатор слота |
| track_config_id | FK → TrackConfig | Конфигурация трассы |
| marshal_id | FK → Marshal | Назначенный маршал |
| start_at | datetime (UTC) | Дата и время старта в UTC; **хранится в UTC**, **источник истины — сервер**. Клиент отображает в **локальной зоне центра**, но право/тип отмены (правило 2 часов) вычисляет сервер (R-021) |
| total_seats | int | Всего мест (≤ capacity_cap конфигурации) |
| free_seats | int | Свободно мест (расчётное/денормализованное) |
| free_rental_equipment | int | Свободно прокатных комплектов экипировки (шлем + подшлемник) |
| price | money (RUB) | Цена заезда за одно место (валюта — рубли) |
| rental_price | money (RUB) | **Отдельный тариф проката** за один комплект экипировки (отдельно от `price`); своя экипировка бесплатна. Итог брони = `price × seats_count + rental_price × rental_count` (R-010) |
| meeting_point | string | **Место сбора** (адрес/ориентир) — обязательное; текстовый блок на SCR-003/SCR-006 |
| meeting_point_lat | float | Широта точки сбора — пин на карте |
| meeting_point_lng | float | Долгота точки сбора — пин на карте |
| status | enum (`active`/`full`/`cancelled`/`started`) | Статус слота |
| cancel_reason | string? | Причина отмены (если status = cancelled) |

> **Предзаполняется существующей инфраструктурой.** Приложение не создаёт и не редактирует слоты —
> только потребляет их через API. `free_seats` и `free_rental_equipment` обновляются атомарно
> при бронировании на стороне бэкенда (NFR-8).

> **Отмена слота центром.** При отмене слота (`status → cancelled`) все связанные брони
> переходят в `club_cancelled` — **«Отменён центром»** (отмена не по инициативе клиента, R-008).

---

### Booking (Бронь / Запись) — **ИЗМЕНЯЕТСЯ ПРИЛОЖЕНИЕМ**

| Атрибут | Тип | Описание |
| :-- | :-- | :-- |
| id | UUID (PK) | Идентификатор брони |
| slot_id | FK → Slot | Слот, на который записались |
| client_id | FK → Client | Клиент, оформивший бронь |
| seats_count | int | Число мест в брони (1–3: себя + до 2 гостей). Только агрегат, без сущности BookingSeat (R-013) |
| rental_count | int | Сколько из мест — на прокатной экипировке (0..seats_count). Агрегат, без BookingSeat (R-013) |
| status | enum (`active`/`cancelled`/`late_cancel`/`club_cancelled`/`completed`) | Статус брони. `club_cancelled` — **«Отменён центром»**: возникает, когда слот отменён центром (`Slot.status = cancelled`); клиент видит бронь как отменённую не по своей инициативе (R-008) |
| price_total | money (RUB), read-only | **Итоговая цена, рассчитанная и возвращаемая сервером** (read-only). Клиент использует её как есть и **не пересчитывает**; `price`/`rental_price` лежат в связанном `slot`. Цена **фиксируется на момент брони** (R-005, R-010) |
| created_at | datetime | Время создания брони |
| cancelled_at | datetime? | Время отмены (если была) |

> **Агрегированные данные по гостям.** По гостям хранятся **только** агрегаты `seats_count` и
> `rental_count` — отдельной сущности `BookingSeat` (вариант/имя по каждому гостю) в скоупе
> **нет** (R-013). Вариант экипировки выводится из `rental_count` (прокатные места) и
> `seats_count − rental_count` (своя экипировка).

> **Итоговая цена `price_total`** рассчитывается на сервере и приходит в ответе API как read-only
> (R-005). Клиент **не пересчитывает** её, а отображает; исходные тарифы `price`/`rental_price`
> лежат в связанном `slot`. Расчёт сервера: `price × seats_count + rental_price × rental_count`,
> валюта — рубли; **цена фиксируется на момент создания брони** (R-010). SCR-005/SCR-006 показывают
> `price_total`.

> **«Прошедшая» — не хранимый статус.** Бейдж «Прошедшая» (SCR-006) и группа «Прошедшие»
> (SCR-005) — **производное отображение** по `Slot.start_at` в прошлом, а не значение
> `Booking.status`. Статус остаётся `active`/`cancelled`/`late_cancel`; «прошедшесть»
> вычисляется из времени старта.

## ERD

```mermaid
erDiagram
    Client ||--o{ Booking : "создаёт"
    Slot   ||--o{ Booking : "содержит"
    TrackConfig  ||--o{ Slot    : "определяет"
    Marshal ||--o{ Slot : "ведёт"

    Client {
        uuid id PK
        string name
        string phone
        datetime created_at
        datetime deleted_at
    }
    TrackConfig {
        uuid id PK
        string name
        string description
        enum type
        int capacity_cap
        int duration_min
    }
    Marshal {
        uuid id PK
        string name
        string photo_url
    }
    Slot {
        uuid id PK
        uuid track_config_id FK

        int free_seats
        int free_rental_equipment
        money price
        money rental_price
        string meeting_point
        float meeting_point_lat
        float meeting_point_lng
        enum status
        string cancel_reason
    }
    Booking {
        uuid id PK
        uuid slot_id FK
        uuid client_id FK
        int seats_count
        int rental_count
        enum status
        money price_total
        datetime created_at
        datetime cancelled_at
    }
