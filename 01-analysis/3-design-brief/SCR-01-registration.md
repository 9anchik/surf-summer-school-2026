# SCR-01: Регистрация и Вход — Постановка на дизайн

## 1. Метаданные

| Поле | Значение |
| :-- | :-- |
| **ID экрана** | SCR-01 |
| **Название** | Регистрация и Вход |
| **Роль пользователя** | Клиент |
| **Тип экрана** | Онбординг / Модальный (fullscreen) |
| **Связанные US** | US-1 |
| **Связанные UC** | UC-4 |
| **Ключевые FR** | FR-1, FR-2, FR-43 |
| **Связанные NFR** | NFR-3, NFR-18, NFR-19, NFR-20 |

## 2. Цель экрана и JTBD

- **JTBD:** Когда я впервые открываю приложение или моя сессия истекла, я хочу быстро подтвердить свой номер телефона без пароля, чтобы сразу увидеть расписание и не тратить время на регистрацию.
- **Бизнес-цель:** Снизить барьер входа до нуля → увеличить долю самостоятельных онлайн-записей (M-1).
- **Главный вопрос, на который экран отвечает:** «Как мне быстро попасть внутрь?»

## 3. Пользователь и контекст использования

- **Окружение:** Где угодно — дома на диване, в дороге, или прямо у трассы (спонтанное решение записаться).
- **Эмоция:** Нетерпение, желание быстрее увидеть слоты.
- **Устройство и хват:** Mobile-first, одной рукой, большой палец.
- **Время на экране:** 15–30 секунд.

## 4. Информационная архитектура и контент

### 4.1. Иерархия информации
1. **Логотип / название центра** (брендинг).
2. **Поле ввода телефона** (ключевой элемент).
3. **Кнопка CTA** «Получить код».
4. **Ссылка** «Политика конфиденциальности» (NFR-20, согласие на ПДн).

### 4.2. Поля и данные

| Поле | Обязательность | Формат / Лимиты | Примечание |
| :-- | :-- | :-- | :-- |
| Номер телефона | Обязательно | Маска `+7 (___) ___-__-__`, валидация формата | Автоподстановка из контактов (опц.) |
| OTP-код | Обязательно | 4 или 6 цифр (определяется бэкендом) | Цифровая клавиатура |
| Таймер повторной отправки | Обязательно | Формат `00:30` → `Отправить повторно` | Активен после истечения |

### 4.3. Ограничения контента
- Номер телефона — только в международном формате (+7…).
- Текст согласия на обработку ПДн — короткий, с ссылкой на политику.

## 5. Пользовательские сценарии

### 5.1. Основные потоки
1. Пользователь открывает приложение → видит экран ввода телефона.
2. Вводит номер → тапает «Получить код».
3. Переходит на экран ввода OTP (с таймером).
4. Вводит код из SMS (или использует системный авто-ввод).
5. Авторизуется → переход на SCR-02 (Главная).

### 5.2. Альтернативные потоки
- **A1. Повторная отправка:** Таймер истёк → пользователь тапает «Отправить повторно» → новый код, предыдущий инвалидируется.
- **A2. У пользователя уже есть аккаунт:** После ввода того же номера сервер возвращает токены, профиль подтягивается автоматически.
- **A3. Новый пользователь:** Сервер создаёт минимальный профиль → приложение запрашивает имя (опц. модалка) или подставляет из контактов.

### 5.3. Точки принятия решений
- На экране OTP — только ввод кода, никаких лишних действий.

## 6. Состояния экрана и компонентов

### 6.1. Загрузка
- **Спиннер внутри CTA** при отправке запроса (кнопка блокируется).
- **Скелетон не нужен** — экран статичный.

### 6.2. Пустые состояния
- Не применимо.

### 6.3. Ошибки
- **Неверный код:** Поле OTP подсвечивается красным, текст «Неверный код, попробуйте ещё раз». Счётчик попыток.
- **Превышен лимит попыток (NFR-19):** Блокировка на N минут, текст «Слишком много попыток. Повторите через 10 минут».
- **Rate limit на отправку (429):** Текст «Слишком часто. Подождите N секунд».
- **SMS не дошло:** Кнопка «Отправить повторно» активна после таймера.

### 6.4. Офлайн-режим
- Баннер «Нет подключения». Кнопка CTA неактивна.

### 6.5. Ограничения
- Лимит попыток ввода кода (≤5) — см. NFR-19.
- Лимит отправки SMS (1 в 30–60 сек, 5 в час).

## 7. Навигация и переходы

- **Вход на экран:** Первый запуск приложения / истечение refresh-токена (NFR-18).
- **Выход:** Системный Back (если есть куда) / Swipe.
- **Поведение Back:** На экране OTP — возврат к вводу телефона (код инвалидируется). На экране телефона — выход из приложения.
- **Deep Links:** Не применимо.

## 8. Визуальный язык и настроение

- **Вайб:** Минимализм, скорость, драйв. Без лишних украшений.
- **Цветовая семантика:**
  - 🟢 Успех: код принят, короткий green-flash.
  - 🔴 Ошибка: красная подсветка поля, мягкий shake-эффект.
- **Типографика:** Крупный заголовок «Войти», моноширинный шрифт для OTP (чтобы цифры легко читались).
- **Референсы:** Экраны входа в Uber, Deliveroo — максимально просто.

## 9. Технические ограничения

- **Тач-зоны:** Поле ввода и CTA — ≥44pt.
- **Safe Area:** Учесть «чёлку» и Home Indicator.
- **Клавиатуры:**
  - Телефон: цифровая с `+`, `()`, `-`.
  - OTP: только цифры, авто-переход между полями (если 4 отдельных поля) или одно поле с авто-вводом из SMS (iOS `textContentType = .oneTimeCode`).
- **Контраст:** WCAG AA.
- **Системные шрифты:** Dynamic Type поддерживается.

## 10. Анимации и микроинтеракции

- **Переход телефон → OTP:** Плавный slide-up, поле телефона «превращается» в поле OTP.
- **Состояние CTA:** Press state (затемнение), при загрузке — спиннер внутри.
- **Ошибка OTP:** Лёгкий shake поля + красная подсветка.
- **Таймер:** Плавная анимация обратного отсчёта.

## 11. Критерии приёмки

- [ ] Проработаны состояния: ввод телефона, отправка, ввод OTP, успех, все виды ошибок.
- [ ] Учтён системный авто-ввод OTP из SMS (iOS/Android).
- [ ] Продуманы клавиатуры для обоих экранов.
- [ ] Соблюдены тач-зоны и контраст.
- [ ] Есть ссылка на политику конфиденциальности (NFR-20).
- [ ] Учтены rate-limit и блокировки (NFR-19).

## 12. Артефакты на выходе

1. Два экрана: ввод телефона + ввод OTP (все состояния).
2. Компоненты: поле ввода с маской, OTP-поле, таймер, кнопка с состояниями.
3. Прототип: Happy Path + повторная отправка + неверный код.
4. Спецификация: маски, форматы, тексты ошибок (копируемые строки).
5. Ассеты: логотип, иконка SMS (опц.).
