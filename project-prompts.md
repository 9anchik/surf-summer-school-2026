# Каталог промптов для учебного проекта Apex Carting

Документ содержит структурированный набор промптов, которые можно использовать для сопровождения учебного проекта: от бизнес-анализа и требований до проектирования API, реализации backend-сервиса, тестирования и код-ревью.

Промпты адаптированы под последовательную работу с проектом и сформулированы так, чтобы результат можно было сохранять в отдельные `.md`, `.yaml`, `.sql` или Go-файлы.

## 6. Бронирования

### 6.1. Создание брони

```text
Реализуй POST /api/v1/bookings.

Создание брони должно выполняться в транзакции и защищать от овербукинга.

Логика:
1. Получить user_id из JWT.
2. Проверить наличие Idempotency-Key.
3. Открыть транзакцию.
4. Проверить, не существует ли уже бронь с таким Idempotency-Key для пользователя.
5. Заблокировать слот через SELECT ... FOR UPDATE.
6. Проверить, что слот активен.
7. Проверить количество мест: от 1 до 3.
8. Проверить свободные места.
9. Отдельно проверить свободную прокатную экипировку.
10. Рассчитать price_total на сервере.
11. Создать booking.
12. Создать booking_seats.
13. Обновить счётчики слота.
14. Выполнить commit.

Верни корректные ошибки: slot_full, rental_unavailable, slot_cancelled, invalid_request, missing_idempotency_key.
```

### 6.2. Список моих бронирований

```text
Реализуй GET /api/v1/bookings.

Endpoint должен возвращать только бронирования текущего пользователя. user_id берётся из JWT middleware.

Добавь пагинацию limit/offset. В ответ включи данные брони и краткую информацию о слоте: время старта, время прибытия, конфигурация трассы, маршал, адрес, место сбора, статус слота и причину отмены, если есть.
```

### 6.3. Отмена брони

```text
Реализуй POST /api/v1/bookings/{id}/cancel.

Отмена должна выполняться только для брони текущего пользователя.

Правила:
- отменяется только вся бронь целиком;
- если до старта больше или ровно 3 часа, статус становится cancelled, места и прокатная экипировка освобождаются;
- если до старта меньше 3 часов, статус становится late_cancel, места и экипировка не освобождаются;
- если заезд уже начался, отмена запрещена;
- если бронь уже отменена, вернуть ошибку.

Сервер является источником истины по времени.
```

### 6.4. Детали одной брони

```text
Реализуй GET /api/v1/bookings/{id}.

Endpoint должен возвращать только бронь текущего пользователя. В ответ включи:
- основные данные брони;
- список выбранной экипировки по местам;
- данные связанного слота;
- статус слота и причину отмены, если слот отменён центром.
```

### 8.3. Исправление ошибки в create booking

```text
Исправь метод Create в bookings repository.

Проблема: при проверке существующей брони по Idempotency-Key любая ошибка SELECT обрабатывается как отсутствие записи. Нужно отличать pgx.ErrNoRows от реальных ошибок БД.

Исправь метод так, чтобы:
- при найденной записи возвращалась существующая бронь;
- при pgx.ErrNoRows выполнение продолжалось;
- при любой другой ошибке метод возвращал ошибку;
- при поиске слота pgx.ErrNoRows маппился в ErrSlotNotFound, а остальные ошибки возвращались как есть.
```

### 8.4. Исправление ошибки в FindOrCreateUser

```text
Исправь метод FindOrCreateUser в auth repository.

Проблема: если SELECT пользователя по телефону завершился ошибкой БД, метод всё равно пытается создать нового пользователя.

Исправь метод так, чтобы:
- при найденном пользователе возвращался его id;
- если передано имя, оно обновлялось, а ошибка UPDATE не игнорировалась;
- при pgx.ErrNoRows создавался новый пользователь;
- при любой другой ошибке SELECT метод возвращал ошибку и не выполнял INSERT.
```

## 11. Универсальные правила для дальнейших промптов

### 11.1. Требование к формату ответа

```text
Отвечай структурированно. Если предлагаешь реализацию, разделяй ответ по файлам и указывай путь к каждому файлу.

Если предлагаешь исправление бага, сначала кратко опиши проблему, затем покажи исправленный код.

Если создаёшь документацию, оформляй её в markdown и сохраняй в отдельный файл.
```

### 11.2. Требование к учебному MVP

```text
Проект является учебным MVP. Не усложняй реализацию без необходимости.

Предпочитай простые решения:
- layered architecture вместо сложной clean architecture;
- dev-заглушки для SMS и push;
- один PostgreSQL;
- простые unit-тесты через fake repository;
- минимум внешних зависимостей.

При этом не упрощай критичные бизнес-правила: защита от овербукинга, транзакционность бронирования, доступ только к своим данным и корректная обработка ошибок должны быть реализованы надёжно.
```
